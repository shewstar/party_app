Builder prompt

Build a mobile-first web app for a private bucks party with around 12 concurrent users in a single shared party room. The app should feel minimalist, fast, touch-friendly, and simple to use while people are socializing and drinking. Optimize for phones first, especially screens around 360–390px wide, with large tap targets, minimal typing, and clear visual hierarchy. [web:44][web:46][web:41]

Product goal:
Create a lightweight shared party utility where users can log drinks, see their current drink tally and estimated BAC, vote on proposed rules or ideas, manage games and scores, and view live leaderboards.

Core first-time user flow:
1. User opens app on phone.
2. User enters their name.
3. User lands on the home page.
4. Home page has a clean minimalist layout with large touch-friendly buttons.
5. The largest and top-most button is “Add a Drink”.
6. Below it are smaller buttons: “Games”, “Leaderboards”, “Vote”, and “Settings”.
7. On the home page, also display the user’s current drinks number and current estimated BAC near the top so this information is visible immediately.
8. Lower on the home page, show the current drinks leaderboard.
9. If the user scrolls further, show a list of items that currently have net positive “for” votes from the vote system.

Home page requirements:
- Mobile-first single-column layout.
- “Add a Drink” must be the dominant visual action.
- Show current drink tally and current estimated BAC as summary cards or compact info blocks.
- Show current drinks leaderboard lower on the screen.
- Show a scrollable section of currently active or positively voted “for” items below the leaderboard.
- Keep the design minimalist, with lots of breathing room, simple typography, subtle borders, and no clutter. [web:44][web:46]

Add a Drink flow:
When the user taps “Add a Drink”, open a screen or sheet with three large buttons:
- Beer
- Wine
- Spirits

From there, guide the user through a logical, low-friction flow to choose drink details and log the drink.

Drink logging should:
- Increase the user’s personal drink tally.
- Increase or recalculate the user’s estimated BAC.
- Store each drink with timestamp.
- Support quick presets to minimize typing.

Suggested drink flows:
- Beer: choose common sizes and strengths.
- Wine: choose glass size and ABV or standard preset.
- Spirits: choose shot or mixed drink size and spirit strength.

BAC requirements:
- BAC must be clearly labeled as an estimate only.
- BAC must not be framed as medical, legal, or driving advice.
- Add a visible disclaimer that users should not use the app to decide whether it is safe or legal to drive.
- Use profile inputs like weight, sex, and time of first drink to estimate BAC.
- Use standard drink calculations and elapsed time in the estimate. [web:33][web:25][web:36]

Games section:
The Games page should:
- Show a list of games.
- Allow adding a new game.
- Allow selecting who is playing from the party members.
- Allow entering and updating scores.
- Support simple party game scorekeeping.
- Feed game results into leaderboards.

Examples of game support:
- Beer pong
- Pool
- Darts
- Trivia
- Any custom named game

Leaderboards section:
The Leaderboards page should show live rankings for:
- Drinks consumed
- Estimated BAC
- Games / wins / score totals
- Optional overall points if there is a sensible combined model

Vote section:
The Vote page should:
- Have a text input to submit a new rule, challenge, or proposal.
- Show the current vote list.
- Let each item be voted “for” or “against”.
- Show current totals clearly.
- Make it very easy to vote quickly on mobile.
- Surface the positively voted items back onto the home page lower down as a scrolling section.

Settings section:
The Settings page should allow the user to set or edit:
- Name
- Weight
- Sex or BAC profile inputs
- Preferred units / drink presets
- Optional profile picture or avatar
- Reset or clear personal drink history
- Session preferences or party settings if needed

Nice-to-have features:
- Profile pictures for each member.
- Avatar circles beside names in leaderboards, votes, and games.
- Lightweight real-time sync for all users in the same room.
- Optimistic UI updates so votes and drink logs feel instant.

Design direction:
- Minimalist and modern.
- Mobile-first, not a desktop dashboard crammed onto a phone.
- Large primary action on the home screen.
- Smaller secondary actions below it.
- Strong spacing and obvious hierarchy.
- Clean cards, subtle borders, soft background, restrained color palette.
- No enterprise/admin-panel feel.
- No visual clutter, gimmicks, or over-designed gradients. [web:44][web:40][web:46]

Technical requirements:
- Build as a web app, not a native app.
- Design for about 12 simultaneous users in one shared event.
- Prefer a simple, reliable architecture.
- Support one shared party room/session.
- Data model should support users, drinks, drink entries, BAC profile data, votes, games, scores, and leaderboard calculations.
- Profile pictures should be optional.

Suggested implementation approach:
- Use a mobile-first frontend.
- Use a simple backend or hosted realtime database.
- Prioritize the first-time phone flow, homepage, drink logging flow, voting flow, and live leaderboard updates before advanced polish.
- Reduce typing wherever possible by using presets, chips, segmented controls, and buttons. [web:46]

Suggested tech stack:
- React or Next.js frontend
- Supabase for auth-lite session data, database, realtime updates, and optional profile image storage
- Simple room/session model instead of complex accounts
- Responsive component system optimized for mobile

Suggested data model:
- party_rooms
- users
- drink_entries
- vote_items
- vote_responses
- games
- game_players
- game_scores
- leaderboard_views or computed queries

Important UX rules:
- The user should understand the home screen in under 2 seconds.
- “Add a Drink” must always be the clearest next action. [web:44][web:46]
- Keep common tasks to 1–2 taps where possible.
- Use progressive disclosure and avoid long forms. [web:44]
- Make vote interactions and drink logging extremely fast.
- Keep all content readable and usable on a phone in one hand.

Deliverables requested from the AI coder:
1. App architecture and feature breakdown
2. Screen-by-screen UI plan
3. Component list
4. Database schema
5. Realtime event flow
6. MVP build plan
7. Actual implementation of the app

Final product principle:
Favor speed, clarity, and low-friction mobile interactions over feature richness. This is a social event web app for real-time use during a live bucks party, so it must feel simple, fast, and obvious. [web:43][web:46]
